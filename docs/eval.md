# 人工评测说明

用于检查“猪猪产品毒舌官”的评审报告是否有真实产品判断，而不是模板化总结。

## 评测重点

- `goDecision` 是否合理：`CONTINUE`、`PIVOT`、`PAUSE` 不能都给同一种。
- `goDecisionReason` 是否指出最大风险。
- 毒打指数是否体现重构程度，产品定位评分是否体现切口清晰度。
- 成功指标是否可观察，验证计划是否可执行。
- 开发 Prompt 是否能直接交给 Codex/Cursor。

## 样例 1：校园搭子小程序

输入：

```text
我想做一个校园搭子小程序，支持饭搭子、学习搭子、运动搭子匹配，还有校园吐槽广场。
```

预期观察：

- 应指出冷启动、社区冗余和多场景同时起步的风险。
- `goDecision` 应倾向 `PAUSE` 或 `PIVOT`。
- MVP 建议应砍掉吐槽广场，先验证一个高频搭子场景。

## 样例 2：AI 产品评审工具

输入：

```text
我想做一个 AI 产品评审工具，用户输入产品想法后，系统输出伪需求风险、MVP 范围和给 Cursor 的开发 Prompt。
```

预期观察：

- 应识别闭环较清晰，目标用户可以是独立开发者、小团队或课程项目开发者。
- `goDecision` 应倾向 `CONTINUE`。
- 仍应建议收缩 MVP，强调报告质量、复制 Prompt 和历史记录，不应扩展成自由聊天平台。

## 样例 3：甲方校园服务小程序

输入：

```text
甲方想做一个校园服务小程序，包含活动报名、帖子社区、公告轮播、用户管理、数据统计、举报审核和移动业务宣传。
```

预期观察：

- 应指出范围过大、交付边界不清、验收风险高。
- `goDecision` 应倾向 `PIVOT`。
- 验证计划应要求先定义一个可验收业务目标，而不是直接开发全量模块。
